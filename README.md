# Report Catalog — Data Lineage Explorer

A self-contained report and data lineage catalog. Non-technical users double-click `index.html`. Developers edit markdown files and run `bundle.py` to regenerate.

---

## Quick Start

1. **View the catalog:** Open `index.html` in any browser (no server needed)
2. **Update docs:** Edit any `.md` file in `sources/`, `processes/`, or `reports/`
3. **Update lineage:** Edit the `depends_on:` list in the relevant `.md` file (NOT `dependencies.yaml` — that file is now generated)
4. **Regenerate:** Run `python bundle.py`

To bulk-create placeholders instead of hand-writing them, see **[Generator scripts](#generator-scripts)** (`scripts/introspect_db.py` for databases, `scripts/scan_ssrs.py` for SSRS reports).

## Markdown Frontmatter

Every `.md` file starts with a YAML frontmatter block:

```yaml
---
id: tbl_orders           # required — unique; referenced by other files' depends_on
label: dbo.Orders        # display name
subtype: Table           # see types below
owner: DBA Team
updated: 2026-04-01
criticality: High        # Critical | High | Medium | Low (optional)
depends_on:              # upstream/parent asset IDs (optional)
  - etl_sales_hr_load
---

## Documentation goes here in Markdown

Tables, code blocks, bold, lists, headers — all supported.
```

### Criticality

Add `criticality: Critical | High | Medium | Low` to any file. It renders as a
colored corner dot on the node and a bold colored pill in the detail panel, and
is searchable (type "critical" in the search box). Critical = red, High =
orange, Medium = amber, Low = slate.

### Asset Types (`subtype`)

| Category | Subtypes |
|---|---|
| **Sources** | `Table`, `View`, `REST API`, `File/CSV` |
| **Processes** | `Python`, `Stored Proc`, `SQL Script` |
| **Reports** | `SSRS`, `PowerBI` |

Any value is accepted — these are the recognized ones with distinct icons.

### Server folders

Create a folder under `sources/` for each SQL Server. Add `_server.md` with frontmatter for the server node. Create subfolders for each database, add `_database.md`, then one `.md` per table/view.

### Flat sources

Single `.md` files directly in `sources/` (no subfolder) render as standalone nodes — use for APIs, file drops, external data feeds.

## Dependencies

**`dependencies.yaml` is now GENERATED — do not edit it by hand.** Lineage lives
on each asset instead. In every `.md`, list only the asset's **parents**
(upstream IDs); the child listing its parent is enough to draw the edge, so you
never declare the same edge twice:

```yaml
---
id: proc_sales_etl
depends_on:            # this process reads from these upstream assets
  - tbl_orders
  - tbl_customers
  - src_crm_api
---
```

On `python bundle.py`, every file's `depends_on` (plus any tool-inferred
`depends_on_auto`, see below) is collected into edges and written out to
`dependencies.yaml` for inspection. IDs must match the `id` field of a real
asset.

There are two dependency keys:

| Key | Who owns it | Survives re-scan? |
|---|---|---|
| `depends_on` | **You** — hand-curated lineage | Yes, always preserved |
| `depends_on_auto` | Tools (`scan_ssrs.py`) — inferred lineage | Overwritten on `--refresh` |

Both contribute edges; keeping them separate means re-running a generator never
clobbers your manual edits.

**Validate without regenerating** (checks every `depends_on` ID resolves):

```bash
python bundle.py --check
```

## Generator scripts

Instead of hand-creating placeholder `.md` files, generate them. Both scripts
are **safe to re-run** — existing files are never overwritten (only the
tool-owned `<!-- AUTO:... -->` block of a file is refreshed, and only when you
ask). Run `python bundle.py` afterward to fold the new files into the catalog.

### `scripts/introspect_db.py` — database placeholders

Connects to SQL Server (pyodbc, Windows/Integrated auth by default) and writes
`_server.md`, `_database.md`, and one `<schema>_<object>.md` per table — each
prefilled with column grid, row count, and PK info.

```bash
pip install pyodbc      # + Microsoft "ODBC Driver 18 for SQL Server"

python scripts/introspect_db.py --server PROD-SQL-01 \
    --databases SalesOLTP HumanResources [--include-views]
```

Useful flags: `--host` (network name if it differs from the folder name),
`--sql-auth` (use `CATALOG_DB_USER`/`CATALOG_DB_PASSWORD` env vars instead of
Windows auth), `--dry-run`, `--refresh` (re-pull column grids, keep your prose).

**Permissions:** reads only `sys.*` catalog views — never your table data. SQL
Server metadata visibility means a login sees catalog metadata only for objects
it has a permission on, so the script self-limits to your access:

| Your access | What to grant | You get |
|---|---|---|
| A few views | nothing extra (existing `SELECT`) | placeholders for just those views |
| Whole DB, no data | `GRANT VIEW DEFINITION` (db-scoped) | all object metadata, no data |
| Whole DB | `db_datareader` role | everything incl. reliable row counts |

Row counts come from `sys.partitions` (a catalog view), so **no `VIEW DATABASE
STATE` is required**; if it's blocked the script continues with counts = 0. A
metadata error on one database is logged and skipped, so mixing high- and
low-access databases in one run is safe.

### `scripts/scan_ssrs.py` — SSRS report placeholders

Points at a folder of `.rdl` files and writes `reports/<name>.md` for each,
extracting datasets, data sources/databases, and the tables their SQL touches.
Referenced tables are auto-matched to catalog IDs and written to
`depends_on_auto`.

```bash
python scripts/scan_ssrs.py --rdl-dir /srv/ssrs/reports [--owner "BI Team"]
```

The auto-match is a **first pass**. After it, edit `depends_on` by hand for the
authoritative lineage — your edits are kept. Re-pull auto data later with
`--refresh` (all reports) or `--only "Report Name"` (just one). Unmatched table
references are listed in the file's AUTO section so you can wire them manually.

## Regenerating

```bash
python bundle.py
```

Requires Python 3.8+. PyYAML is recommended but optional:
```bash
pip install pyyaml
```

Without PyYAML, a minimal fallback parser handles simple frontmatter (no nested YAML, no lists in frontmatter).

## UI Features

- **Click any node** — view documentation, upstream/downstream deps
- **Click dep chips** — navigate to connected assets
- **Search** — filters by name, subtype, owner, or documentation content; connected context nodes stay visible
- **Impact Analysis** — toggle in header; click a node to see blast radius (how many downstream assets are affected if it fails)
- **Collapse / expand databases** — databases start collapsed (showing just a header with the table count). Click the ▾/▸ toggle area on the left of a database header — or anywhere on a collapsed database's row — to expand or collapse it. Use **Collapse all / Expand all** in the header to toggle every database at once. While collapsed, a database's table-level lineage is aggregated onto the database node, so edges and impact analysis still work.
- **Esc** — close detail panel
- Works entirely offline once loaded (fonts load from Google CDN on first open)
